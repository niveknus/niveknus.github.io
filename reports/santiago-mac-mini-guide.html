🔐 Reaching the mini: SSH vs VNC

Once it's headless, you get in two ways. Both ride over Tailscale (install it on the mini and your laptop/phone, sign into the same account — now they're on a private network and you use the mini's Tailscale IP from anywhere, no router config).

⌨️ SSH a text terminal

A command line on the mini. Fast, lightweight, works on the worst connection.

ssh santiago@<mini-tailscale-ip>
  • Running commands, editing files, checking logs
  • Starting/stopping launchd jobs
  • Attaching to tmux sessions
→ Use for 95% of admin. It's your daily door.

🖱️ VNC the actual desktop

macOS Screen Sharing (built in) — see and click the mini's real desktop. Heavier, needs a decent connection.

# Mac: Finder → Go → Connect to Server
vnc://<mini-tailscale-ip>
# Phone: RealVNC Viewer app
  • Clicking through GUI app setup (first-run dialogs)
  • Pairing apps via QR code (e.g. OpenMessage)
  • Anything that pops a window SSH can't reach
→ Use only when you need to click something.
🧭 Rule of thumbSSH by default, VNC when a mouse is required. You'll live in SSH and only reach for Screen Sharing a few times a month.

🛰️ Pro move: make every session know where it is

Here's a trick Kevin uses that's worth stealing. Claude Code can run a hook at the start of every session — a script whose output gets injected into Claude's context. He uses it so Claude instantly knows which machine it's on and how you connected, then adjusts its behavior (e.g. the mini doesn't auto-push to git, so it reminds itself to push manually).

How it works

A SessionStart hook in .claude/settings.json runs a small bash script. The script:

  1. Reads the hostname → maps it to a friendly name ("Mac mini" vs "MacBook Air"). The same script works on both machines because it detects the host live.
  2. Walks the parent-process chain looking for remote-control. If it finds it, you came in from the phone; if not, you're on a direct terminal/SSH session.
  3. Picks the right git behavior for that host and tells Claude.

Wire it up

// .claude/settings.json
"hooks": {
  "SessionStart": [{ "hooks": [{
    "type": "command",
    "command": "bash \"$CLAUDE_PROJECT_DIR/scripts/detect-context.sh\"",
    "timeout": 10
  }]}]
}

And the script prints JSON that Claude treats as context:

host="$(hostname -s)"
case "$host" in
  *Mac-mini*)  device="Mac mini (server)" ;;
  *Air*)       device="MacBook Air (laptop)" ;;
esac
# walk parents for 'remote-control' to detect phone vs SSH...
ctx="SESSION CONTEXT — Device: $device · Transport: $transport"
printf '{"hookSpecificOutput":{"hookEventName":"SessionStart",'
printf '"additionalContext":"%s"}}\n' "$ctx"
Why bother?Once Claude knows it's on the always-on box reached from your phone, it stops making laptop assumptions — it remembers to push changes, knows which tokens are local, and talks to you like the mobile session it is. Small hook, big quality-of-life.

♻️ launchd in 60 seconds

The babysitter. Drop a .plist file in ~/Library/LaunchAgents/ and macOS runs it your way.

You want…plist key
Start at loginRunAtLoad
Restart if it diesKeepAlive
Don't crash-loopThrottleInterval (e.g. 30s)
Run every N secondsStartInterval
Run at a clock timeStartCalendarInterval

The four commands you'll actually use

(501 is your user id — check with id -u. The label is the plist's Label string.)

launchctl bootstrap gui/501 ~/Library/LaunchAgents/<label>.plist  # start
launchctl bootout    gui/501/<label>                            # stop
launchctl kickstart -k gui/501/<label>                          # restart
launchctl list | grep <label>                                   # status (PID + exit code)

How it works

  • claude remote-control registers an environment (Kevin names his "Mac mini") with Anthropic over outbound HTTPS only — no open ports, no Tailscale needed for this part.
  • In the phone's Code tab you tap that environment to spawn a fresh session, or tap a numbered one ("Mac mini · 1 of 32") to resume a chat. Up to 32 at once, all running on the mini.
  • Spawn mode same-dir means every session opens in the same working folder.

Set it up (auto-start, self-healing)

1 · Wrapper script ~/mini-scripts/claude-rc.sh

#!/bin/bash
set -uo pipefail
export PATH="/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin"
cd "$HOME/projects" || exit 1   # the folder sessions open in
exec claude remote-control --name "Mac mini" --spawn=same-dir

2 · launchd agent com.santiago.claude-remote-control.plist

RunAtLoad + KeepAlive + ThrottleInterval 30, pointing at that script, with PATH and HOME set and log paths under ~/Library/Logs/. Load it:

launchctl bootstrap gui/501 ~/Library/LaunchAgents/com.santiago.claude-remote-control.plist
launchctl list | grep claude-remote-control   # want a PID and a 0

3 · Use it

Open Claude app → Code → tap "Mac mini". You're now in a full Claude Code session running on the mini, from your phone. Build away.

⚠️ Two prerequisites or it won't start
  • Signed into claude.ai (not API): echo $ANTHROPIC_API_KEY is blank.
  • The working folder is trusted: cd ~/projects && claude once, accept the trust dialog, /exit.
If the err log says "requires a claude.ai subscription," it's the first one.

🪟 tmux: the trick that makes "always-on" possible

tmux = terminal multiplexer. The point for us: it keeps a program running after you disconnect. Start something inside tmux, close your laptop, and it's still running on the mini. That's why the Telegram session survives you walking away.

Do thisCommand
Start a named, detached sessiontmux new-session -d -s channels "…"
List running sessionstmux ls
Attach (watch it live)tmux attach -t channels
Detach (leave it running)Ctrl-b then d
Scroll back through outputCtrl-b then [ (then q to quit scroll)
Kill a sessiontmux kill-session -t channels
🚫 The one mistake everyone makesTo leave a session running you detach (Ctrl-b d). If you type exit or hit Ctrl-c, you kill the program inside. Detach ≠ exit.

Handy aliases (drop in ~/.zshrc)

alias channels="tmux attach -t channels"
alias channels-log="tmux capture-pane -t channels -p -S -2000 | less +G"

Now channels drops you into the live assistant session and Ctrl-b d backs out.

💬 Front door #2 · Channels (text your mini)

A Telegram bot becomes a persistent Claude session on the mini. You text it; it reads your files, hits your connectors, and texts back — your always-on assistant.

1 · Make a bot

Message @BotFather on Telegram → /newbot → save the bot token. Then grab your own chat_id (message @userinfobot).

2 · Token where the plugin reads it

Put the token in ~/.claude/channels/telegram/.env.

3 · Wrapper script ~/mini-scripts/claude-channels.sh

#!/bin/bash
set -uo pipefail
export PATH="/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin"
SESSION="channels"
cd "$HOME/projects" || exit 1
if ! tmux has-session -t "$SESSION" 2>/dev/null; then
  tmux new-session -d -s "$SESSION" -c "$HOME/projects" \
    "env -u ANTHROPIC_API_KEY claude --model sonnet \
       --channels plugin:telegram@claude-plugins-official"
fi
while tmux has-session -t "$SESSION" 2>/dev/null; do sleep 30; done

So launchd keeps the script alive, the script keeps the tmux session alive, and tmux keeps Claude alive. Model pinned to sonnet to stretch subscription limits.

4 · launchd agent + lock it down

Same plist shape as Remote Control, pointing at this script. On first message, approve only your own chat_id (allowlist DM policy) so strangers can't talk to your bot. Then verify it's on your subscription:

tmux attach -t channels   # header must say "Claude Max/Pro", NOT "API Usage Billing"
🎁 Connectors come freeBecause it runs as your subscription in the GUI session, your claude.ai connectors (Calendar, Gmail, Todoist, GitHub, Drive…) are already wired in — no per-machine setup. Text "what's on my calendar?" and it just answers.

🩹 The two gotchas that will bite you

🔑 #1 — Start sessions via launchd, never by hand over SSH A claude session launched from an SSH shell can't unlock the login keychain → can't read your claude.ai token → it silently bills your API account. The exact same command under launchd (GUI session) shows "Claude Max." Always restart with launchctl kickstart, not a manual run. This is also why auto-login matters.
🤖 #2 — One Telegram bot = one poller Telegram allows only one message-poller per bot token. If you run any claude command over SSH that loads the telegram plugin (even claude mcp list), it spins up a second poller, steals the lock, and your live bot goes silent. To inspect from SSH: read logs or tmux attach — don't launch a fresh claude. Recovery: attach → /mcp → select telegram ✗ failed → Reconnect.
🐕 Optional watchdogOnce you've used Channels a while, add a tiny launchd job (every 5 min) that checks the session still has a live poller and restarts it if not. Self-healing. Not needed day one.

☁️ The cloud half (NOT on the mini)

Kevin also runs scheduled claude.ai Routines (morning brief, fitness brief, weekly review) and GitHub Actions (dashboard/report deploys). These run on Anthropic's / GitHub's cloud, so they fire even when the mini is off.

The dividing line: anything needing local tokens (his two Gmail accounts, OpenMessage) lives on the mini; anything pure-cloud (read a calendar, write a file via the GitHub connector) is a cloud Routine so it doesn't depend on the mini being up. Start on the mini, graduate the always-available stuff to the cloud later.
